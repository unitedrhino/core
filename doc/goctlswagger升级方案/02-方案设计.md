# 02 - 方案设计

## 1. 方案选型

### 方案对比

| 方案 | 改动量 | 风险 | 输出质量 |
| ---- | ------ | ---- | -------- |
| A. 后置转换（采用） | ~50 行新代码 | 低 | 好 |
| B. 原生重写 | 8+ 文件重写 | 高 | 最好 |
| C. 第三方插件 | 外部依赖 | 高 | 未知 |

### 选择方案 A 的理由

1. `kin-openapi` 库成熟稳定，自动处理所有标准转换
2. 现有 Swagger 2.0 生成代码完全不动，零破坏风险
3. 转换是纯后处理步骤，可随时回退
4. 新增代码量极小，易于维护

## 2. 技术架构

### 转换流程（升级后）

```text
.api 文件
  ↓ parser.Parse()
ApiSpec 结构体
  ↓ 现有生成逻辑（不变）
Swagger 2.0 JSON（内存中）
  ↓ swagger2ToOpenAPI3()  ← 新增步骤
OpenAPI 3.0 JSON
  ↓ os.WriteFile()
写入文件
```

### 核心依赖：kin-openapi

- **库地址**: `github.com/getkin/kin-openapi`
- **版本**: v0.133.0
- **关键包**:
  - `openapi2` — Swagger 2.0 文档结构体 `openapi2.T`
  - `openapi2conv` — 转换函数 `ToV3()`
  - `openapi3` — OpenAPI 3.0 文档结构体（隐式使用）

## 3. Swagger 2.0 → OpenAPI 3.0 关键映射

| Swagger 2.0 | OpenAPI 3.0 |
| ------------ | ----------- |
| `"swagger": "2.0"` | `"openapi": "3.0.3"` |
| `host` + `basePath` + `schemes` | `servers: [{url: "https://host/basePath"}]` |
| `definitions` | `components.schemas` |
| `$ref: "#/definitions/Foo"` | `$ref: "#/components/schemas/Foo"` |
| `parameters` 中 `in: body` | `requestBody.content.application/json.schema` |
| `parameters` 中 `in: formData` | `requestBody.content.multipart/form-data` |
| `produces` / `consumes` | 每个操作的 `content` 对象 |
| `securityDefinitions` | `components.securitySchemes` |
| Response 的 `schema` | Response 的 `content.application/json.schema` |

## 4. 风险分析

### 风险1: 空 description 字段（中等）

当前 response.go 生成空 `Description: ""`，OpenAPI 3.0 要求非空。
**应对**: 转换后遍历补充默认值，或在转换前修复。

### 风险2: formData 参数转换（中等）

旧版 `in: formData` 需转为 `requestBody` + `multipart/form-data`。
**应对**: `kin-openapi` 已内置此逻辑，需实际验证。

### 风险3: 自定义扩展字段（低）

`x-goctl-version` 等扩展字段。
**应对**: `kin-openapi` 保留 extensions，无需处理。

### 风险4: JSON 属性顺序变化（低）

旧版自定义 `MarshalJSON` 保证属性顺序，转换后可能变化。
**应对**: JSON 属性顺序无语义意义，不影响功能。
